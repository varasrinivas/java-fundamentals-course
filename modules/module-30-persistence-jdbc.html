From SQL rows to Java objects

Every backend talks to a database. JDBC is Java's low-level driver API; JPA/Hibernate is the high-level mapping that turns rows into objects. Both rest on one non-negotiable rule: never build SQL with string concatenation.

Concept

JDBC underneath, JPA on top

JDBC's flow: get a Connection from a pooled DataSource, run a PreparedStatement with ? placeholders, read a ResultSet, and map each row to an object. The placeholders aren't a convenience — they're how you stop SQL injection.

JPA/Hibernate maps @Entity classes to tables and generates the SQL for you, including lazy loading and caching. It's productive but leaky — you still need to understand the SQL it emits (watch for the N+1 query problem). Use JDBC (or a thin layer like jOOQ/Spring Data JDBC) when you want control; use JPA when you want speed of development.

Coming from another language

    
What will surprise you

      
Coming from Python
JDBC is Python's DB-API (cursor.execute(sql, params)) — same parameterized-query discipline. JPA/Hibernate ≈ SQLAlchemy ORM: productive object mapping, with the same need to mind the SQL it generates.

      
Coming from JavaScript
JDBC's PreparedStatement is the parameterized query you'd write with pg/mysql2; JPA/Hibernate plays the role of Prisma/TypeORM. The placeholder-not-concatenation rule is identical across all of them.

      
Coming from C#
JDBC ≈ ADO.NET (SqlConnection, SqlCommand with parameters), and JPA/Hibernate ≈ Entity Framework. If you've used Dapper for control or EF for convenience, the JDBC-vs-JPA trade-off will feel familiar.

  






  

    
Exercises

    
Make it stick

      
1 · The injection trap
Why is "...WHERE id = " + userId dangerous, and how does PreparedStatement fix it?
💡 Show hint
String-concatenating user input lets an attacker inject SQL (e.g. 1 OR 1=1). A PreparedStatement sends the SQL and the parameter separately, so the value is always treated as data, never executable SQL.

      
2 · N+1 surprise
With JPA, looping over 100 orders and reading each order's customer fires 101 queries. What's this called, and one way to avoid it?
💡 Show hint
The N+1 query problem — one query for the list, then one per item for the association. Fix it with an eager JOIN FETCH (or a batch/entity-graph) so the customers load in a single query.
