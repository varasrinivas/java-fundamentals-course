Your code's real computer

You don't run Java on your laptop. You run it on a machine that doesn't physically exist — the JVM. Understanding that imaginary computer explains almost everything strange about Java: the compile step, "write once run anywhere," startup lag, and why it gets faster the longer it runs.

Concept

Two compilers and an imaginary machine

In Python or JavaScript, your source is handed more-or-less directly to an interpreter. In Go or C++, it's compiled all the way down to native machine code for your CPU. Java sits in the middle, and that middle is the whole point.

Java is compiled twice. First, javac turns your .java source into bytecode — a compact instruction set for a CPU that was never manufactured. Then the JVM (Java Virtual Machine) loads that bytecode and, while the program runs, a Just-In-Time (JIT) compiler translates the hot parts into real native code on the fly.

Because bytecode targets the imaginary machine and not your CPU, the same .class file runs unchanged on Windows, macOS, Linux, ARM, or x86 — as long as a JVM exists for that platform. That's "write once, run anywhere," and it's a property of bytecode, not magic.

The closest cousin

.NET works almost identically: C# compiles to IL (bytecode), which the CLR runs with its own JIT. If you know the CLR, you already understand the JVM's shape — just swap the names.

Coming from another language

What will surprise you

Coming from Python

There's no "just run the file." You must compile first (javac Hello.java) then run the result (java Hello). CPython also makes bytecode (.pyc), but hides it. Java makes it explicit — and ships that bytecode, not your source.

Coming from JavaScript

V8 already JITs your JS, so the "gets faster as it runs" idea is familiar. The difference: Java has a real, separate compile step with type checking up front. Many bugs that are runtime TypeErrors in JS are caught by javac before the program ever starts.

Coming from Go

Go compiles straight to a single native binary — no VM, instant startup. Java trades that for portability and a world-class JIT: a long-running Java service can hit throughput a static binary struggles to match, at the cost of slower cold starts and a runtime to ship.

Coming from C#

You're basically home already. javac ≈ Roslyn, bytecode ≈ IL, JVM ≈ CLR, JAR ≈ assembly. The biggest day-one differences are tooling (Maven/Gradle vs MSBuild) and that Java's checked exceptions have no C# equivalent.

Exercises

Make it stick

1 · Predict the commands

You have Greeter.java containing a class named Greeter. Write the exact two terminal commands that compile it and then run it. What new file appears after the first command?

💡 Show hint
Compile with javac Greeter.java — this produces Greeter.class (the bytecode). Run with java Greeter (note: no .class extension — you name the class, not the file).

2 · Why is the second run faster?

A teammate benchmarks a tight loop and notices it runs slowly for the first few thousand iterations, then suddenly speeds up — within the same program run. No code changed. What happened?

💡 Show hint
The JVM starts by interpreting bytecode. Once it sees a method is "hot" (called often), the JIT compiler translates it to optimized native code, and subsequent iterations run at near-native speed. This warm-up is why JVM benchmarks discard the first runs.

3 · One JAR, many machines

You build a .jar on your x86 Windows laptop and email it to a colleague on an ARM Mac. It runs without recompiling. In Go, the same trick fails. Explain the difference in one sentence.

💡 Show hint
The JAR contains bytecode for the platform-independent JVM, so any JVM can run it; a Go binary contains native machine code compiled for one specific OS + CPU architecture.
