Program to a contract, not a class

An interface is a promise: “whatever you are, you can do these things.” It's how Java achieves polymorphism without inheritance, lets a class play many roles, and powers everything from Comparable to the Streams API.

Concept

One contract, many implementations

An interface declares method signatures with no bodies (mostly). Any class can implements it, promising to supply those methods. Callers depend on the interface, so they work with any implementation — present or future.

At the call site the JVM uses dynamic dispatch: it looks up the actual object's method at runtime. Prefer interfaces over abstract classes; use an abstract class only when you must share state or partial implementation. A class extends one class but can implement many interfaces.

    

    
Same idea, familiar name
Java interfaces map almost one-to-one onto C# interfaces — down to default methods (C# 8's default interface methods). The Java twist: interfaces can't hold instance state.

Coming from another language

    
What will surprise you

      
Coming from Python
Python leans on duck typing — if it has area(), it works. Java wants the promise declared: a class must say implements Shape, and the compiler checks it. Protocol/ABC is the closest Python analog.

      
Coming from JavaScript
JS has no real interfaces (TypeScript bolts them on, erased at runtime). Java interfaces exist at runtime — you can check instanceof Shape and dispatch on it.

      
Coming from Go
Go interfaces are implicit: satisfy the methods and you implement it. Java is explicit: you must write implements. Trade-off: Go is terser, Java makes the intent and errors clearer.

  






  

    
Exercises

    
Make it stick

      
1 · Class vs interface
When should you reach for an abstract class instead of an interface?
💡 Show hint
When implementations must share state or common code. Interfaces can't hold instance fields. If subclasses need a shared field or a fully written helper method tied to that state, an abstract class fits; otherwise prefer an interface.

      
2 · Many roles
Can one class implement both Comparable and Runnable? Can it extend two classes?
💡 Show hint
Yes to many interfaces — a class can implements Comparable, Runnable. No to two classes: Java has single class inheritance. Interfaces are how Java sidesteps the diamond problem.
