Where your code finds its dependencies

Java's superpower is its library ecosystem — reachable only through a build tool. Maven and Gradle resolve dependency trees, compile, test, and package, all hinging on one old idea you must understand: the classpath.

Concept

Declare dependencies; the tool assembles the classpath

You declare a dependency by coordinates — group : artifact : version — and the build tool downloads it (and its dependencies, transitively) from a repository into a local cache (Maven's ~/.m2). At compile and run time, every jar must be on the classpath: the list of places the JVM searches for classes.

A ClassNotFoundException almost always means a jar is missing from the classpath; a NoSuchMethodError usually means two versions collided and the wrong one won. Build tools exist to make the classpath correct and reproducible so you don't assemble it by hand.

Coming from another language

    
What will surprise you

      
Coming from Python
Maven/Gradle are pip + venv + setuptools rolled into one, with a key win: transitive resolution is automatic. The classpath is roughly the import search path, but assembled per-build rather than per-environment.

      
Coming from JavaScript
pom.xml/build.gradle ≈ package.json, ~/.m2 ≈ node_modules (but shared across projects). Unlike npm's nested trees, Maven flattens to one version per dependency — hence occasional version-conflict resolution.

      
Coming from C#
Maven/Gradle map onto NuGet + MSBuild; coordinates and restore-then-build are familiar. The classpath is the .NET assembly probing path's cousin — the runtime's list of where to find compiled types.

  






  

    
Exercises

    
Make it stick

      
1 · Read the error
At startup you get ClassNotFoundException: com.foo.Bar. What category of problem is this, and where do you look?
💡 Show hint
A classpath problem — the jar containing com.foo.Bar isn't on the runtime classpath. Check that the dependency is declared with the right scope and is actually packaged/passed via -cp.

      
2 · Two versions walk in
Two of your dependencies each pull in a different version of the same library. What decides which one is used, and what can break?
💡 Show hint
The build tool's conflict resolution picks one version (Maven: “nearest wins”). If code compiled against the other version calls a method missing from the winner, you get NoSuchMethodError at runtime.
